How the Alto ‚Üí TDS Workflow Works

A comprehensive overview of the automated deposit registration process

Overview

The Alto ‚Üí TDS workflow is an automated integration that receives tenancy creation events from Alto property management system and automatically registers deposits with The Dispute Service (TDS). The system intelligently routes deposits to either the Legacy TDS API or the new Salesforce-based API based on organization configuration.

Workflow Steps

Step 1: Organization Lookup & API Selection

‚Ä¢ System receives Agency Reference and Branch ID from the webhook or manual trigger
‚Ä¢ Queries the Organization Mapping table using these identifiers
‚Ä¢ Retrieves the organization's tdsProviderPreference setting (either "legacy" or "salesforce")
‚Ä¢ Also retrieves environment setting (development or production) to determine Alto API endpoint
‚Ä¢ This preference determines which TDS API will be used for the deposit submission

Step 2: Fetch Alto Tenancy Data

‚Ä¢ Authenticates with Alto API using OAuth2 client credentials flow
‚Ä¢ Retrieves comprehensive tenancy information including:
  - Tenancy details (ID, start/end dates, rent amount, deposit amount)
  - Property information (address, bedrooms, property type)
  - Landlord details (contact info, address)
  - Tenant information (names, contact details, emails, phone numbers)
‚Ä¢ Makes multiple API calls to Alto endpoints (/tenancies, /properties, /contacts)

Step 3: Data Validation & Enrichment

‚Ä¢ Validates that all required fields are present (deposit amount, landlord, tenants, property address)
‚Ä¢ Checks deposit amount is greater than zero
‚Ä¢ Verifies at least one landlord and one tenant exist
‚Ä¢ Ensures property address has postcode for region lookup
‚Ä¢ If validation fails, creates a "Pending Integration" record with missing field details
‚Ä¢ Pending integrations are polled periodically to retry when data becomes available

Step 4: Postcode Lookup & Region Mapping

‚Ä¢ Extracts property postcode and landlord postcode from Alto data
‚Ä¢ Queries UK government postcode API to determine administrative regions
‚Ä¢ Maps postcodes to TDS-recognized counties/regions (e.g., "South East", "East of England")
‚Ä¢ Region information is required for TDS deposit registration
‚Ä¢ Results are cached for 24 hours to improve performance

Step 5: TDS Payload Preparation & API Selection

‚Ä¢ Transforms Alto data structure into TDS-compatible format
‚Ä¢ Legacy API Format:
  - Uses TDS member ID (numeric, e.g., "1960473")
  - Simpler payload structure with fewer required fields
  - Contact IDs use Alto's numeric identifiers
  - Endpoint: /api/tds/legacy/deposit
‚Ä¢ Salesforce API Format:
  - Uses alphanumeric member codes (e.g., "A02636EW")
  - More detailed payload with additional metadata
  - Person IDs include Alto contact IDs for landlords, sequential IDs for tenants
  - Includes property details (bedrooms, living rooms, furnished status)
  - Endpoint: /api/tds/salesforce/depositcreation
‚Ä¢ Both formats include tenancy reference, deposit amount, rent, dates, and party information

Step 6: TDS Deposit Submission

‚Ä¢ Submits deposit to appropriate TDS API based on organization preference
‚Ä¢ Legacy API: Returns immediate success/failure response with DAN
‚Ä¢ Salesforce API: May return batch ID if processing is asynchronous
  - If batch_id received: Creates pending integration that polls TDS for completion
  - If DAN received immediately: Marks as successful and proceeds
‚Ä¢ Handles API errors and creates appropriate error records
‚Ä¢ Authentication uses API keys or member credentials from organization mapping

Step 7: Integration Record & Audit Trail

‚Ä¢ Creates integration record in Azure Table Storage with unique ID
‚Ä¢ Records include: tenancy ID, DAN, deposit amount, timestamps, processing steps
‚Ä¢ Logs success/failure status and any error messages
‚Ä¢ Stores full workflow result for audit purposes
‚Ä¢ Data is queryable through the Integration Audit Log tab
‚Ä¢ Successful integrations show DAN number for verification

Pending Integration Handling

When the workflow encounters missing data or async processing:

‚Ä¢ Missing Data: Creates PENDING_DATA record with list of missing fields
‚Ä¢ Async TDS Processing: Creates PENDING_DEPOSIT record with batch_id
‚Ä¢ Polling: Background timer checks pending integrations every 5 minutes
‚Ä¢ Retry Logic: Exponential backoff with configurable max attempts (default: 20)
‚Ä¢ Completion: When data/DAN received, updates to COMPLETED and archives
‚Ä¢ Expiration: After max attempts, marks as EXPIRED and archives
